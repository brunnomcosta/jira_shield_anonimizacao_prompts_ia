JIRA SHIELD

Configuracao da extensao Chrome para exportacao LGPD do Jira.

Options

Resumo operacional

A extensao atua como camada de coleta e preparo: le a issue do Jira, busca o ticket Zendesk vinculado, exporta um PDF anonimizado com os dados do titular e gera os arquivos de entrada para o diagnostics.js rodar no CLI Node.js.

Workspace local para Prompt Diagnostico

Opcional. Quando configurado, o plugin anexa trechos reais dos fontes locais ao Prompt Diagnostico, enriquecendo o contexto enviado a LLM. Requer permissao de leitura concedida pelo navegador para cada pasta.

Regra de elegibilidade: fontes ERP / back-end sao sempre incluidas. Fontes mobile / front-end so entram quando o ticket trouxer palavras-chave como Minha Producao, app mobile, celular, tablet ou sinonimos. Os caminhos definidos aqui ficam sincronizados com o .env do projeto — mas o navegador so consegue ler os arquivos apos voce clicar em Vincular permissao de leitura em cada pasta.
Quando disponivel, o plugin preenche automaticamente com a raiz identificada no build. Ao vincular manualmente, o plugin passa a ler e gravar o .env desta pasta — mantendo os caminhos do workspace sincronizados entre a extensao e o CLI Node.js.
Caminho para os fontes do ERP / back-end Protheus. Sincronizado com o .env. A permissao do navegador e concedida separadamente e pode ser revogada a qualquer momento. O modo Prompt Diagnostico + Contexto Fontes exige ao menos um snippet real desta pasta ou da mobile para prosseguir.
Caminho para os fontes do app mobile / front-end. Sincronizado com o .env. So e incluido no Prompt Diagnostico quando o ticket indicar contexto mobile e a permissao do navegador ja estiver vinculada.
Diretorio de includes do ERP (ex.: C:/totvs/includes). Sincronizado com o .env. Ainda nao e usado para montar contexto de fontes, mas e preservado para manter paridade com a configuracao do CLI.

Templates de prompt

Controla o texto de instrucao enviado a LLM ao usar o botao Gerar Documentacao. O template padrao esta em chrome-extension/prompt-templates.js e pode ser sobrescrito ou complementado aqui sem alterar o arquivo fisico.

Importante: esta configuracao afeta apenas o prompt enviado a LLM. As regras de anonimizacao dos dados do titular (CPF, nome, endeco, etc.) sao aplicadas antes, no pipeline de sanitizacao, e nao podem ser alteradas por aqui.
Precedencia de montagem: override do usuario (substitui a base inteira) → template padrao do arquivo (quando nao ha override) → regras complementares (sempre anexadas ao final). O resultado final e mostrado no campo Template final consolidado abaixo.
Indica qual arquivo fisico da extensao define o template padrao do tipo selecionado. Util para saber onde editar caso voce queira alterar o padrao diretamente no codigo.
Estas sao as regras padrao mantidas no arquivo fisico da extensao. Elas nao alteram a anonimização e servem como base para o prompt enviado a LLM.
Substitui completamente o template base do arquivo para este tipo. Deixe vazio para usar o padrao. Use quando o padrao do sistema nao atender ao contexto do seu time.
Instrucoes adicionais que sao sempre anexadas ao final do template base (ou do override). Ideal para especificidades do time: modulo prioritario, nomenclatura interna, nivel de detalhe esperado, etc.
Visualizacao somente leitura do prompt que sera enviado. Os marcadores como {{titulo}} e {{descricao}} sao preenchidos com os dados reais da issue somente no momento em que voce clicar em Gerar Documentacao.

Jira

Obrigatorio. Usado em todas as operacoes da extensao: leitura da issue, busca do ticket Zendesk vinculado, download de anexos e geracao do PDF anonimizado.

URL base da instancia Jira, sem barra final. Usada em todas as chamadas a API REST do Jira. Ex.: https://jiraproducao.totvs.com.br
Nome tecnico do campo customizado da issue Jira que contem o ID do ticket Zendesk vinculado. Usado para localizar automaticamente o ticket do cliente sem precisar abrir o Zendesk. Para confirmar o campo correto da sua instancia, consulte /rest/api/2/field na API do Jira.

Autenticacao Jira

A extensao tenta autenticar na ordem: 1. token pessoal → 2. usuario + senha → 3. sessao ja aberta no navegador. Preencha apenas o metodo que voce usa; os demais podem ficar em branco.

Token de acesso pessoal (PAT). Metodo preferido para Jira 8.14+. Gerado em Perfil → Seguranca → Tokens de acesso pessoal na interface do Jira.
Subpasta relativa dentro da pasta Downloads do sistema. Todos os arquivos exportados (PDF, JSON) sao salvos aqui. Ex.: shield → Downloads/shield/.

Zendesk

Opcional. Quando preenchido, a extensao busca os dados do ticket Zendesk diretamente pela API. Se nao configurado, a extensao tenta recuperar o conteudo via proxy Jira ou pela aba Zendesk ja aberta no navegador — nessa ordem.

Escopo da extensao: exportacao LGPD, busca de dados do titular e geracao de prompt. O diagnostico completo com LLM, leitura de workspace local e analise ERP/mobile continuam sendo feitos pelo CLI Node.js (npm run diagnose).
